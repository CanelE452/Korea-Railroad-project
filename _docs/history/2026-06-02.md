## YOLO26-pose manual GT finetuning — A(padding) vs B(비패딩) 비교 셋업

### 배경
- forklift_raw 실데이터 finetuning 논의 중, truncation(화면 밖 keypoint) 처리 방식이 쟁점.
- 기존 A = padding 방식(`yolo_pose_manual_padded`, 100px reflect)으로 ft 모델만 존재(`yolo26n_pose_v1_ft_manual`).
- B = 비패딩 + visibility=0 마스킹 방식은 데이터셋·모델 모두 부재 → 공정 비교 위해 셋업.

### 핵심 근거 (Ultralytics 동작 검증)
- `ultralytics/data/augment.py:1328-1329` `RandomPerspective.apply_keypoints`: aug 후 화면 밖 keypoint를 `visibility=0`으로 자동 강등.
- YOLO-pose 기본 aug(mosaic=1.0, scale=0.5, translate=0.1)가 매 step truncation을 자동 합성.
- → padding은 (a) 내장 aug와 충돌(살린 v=2가 다시 v=0), (b) 해상도 손해(640+200→640 리사이즈 시 객체 ~24%↓), (c) 추론 시 동일 padding 강제 부담.
- PnP는 6점 이상이면 성립 → 잘린 코너 버려도 6D 복원 가능. 현 파이프라인(`pnp_solver.py`)엔 비패딩이 적합.

### 추가/변경 파일
- **`challenge/data/yolo_pose_manual/`** (신규): 비패딩 manual GT 변환. A와 동일 14 src(`capture*_manual_gt`), `--pad 0 --val-ratio 0`. **219장 정확 일치**, skip 0, 640×480 원본.
  - truncation 분포: 전체 kp의 9.0%가 v=0, 66/219(30.1%) frame에 화면 밖 코너 → A/B 차이 발생 구간.
- **`challenge/yolo_pose/data_manual.yaml`** (신규): train=yolo_pose_manual, val=yolo_pose/val(A와 동일 합성 holdout 1416), flip_idx=[1,0,3,2,5,4,7,6,8].
- **`challenge/yolo_pose/scripts/ft_manual_nopad.sh`** (신규): A 학습 로그에서 추출한 하이퍼파라미터 1:1 복제(SGD lr0=1e-4 lrf=0.01, 60ep, batch16, imgsz640, patience30, mosaic1.0/close_mosaic10/scale0.5/translate0.1/fliplr0.5/hsv/erasing0.4). 차이는 data와 name(`yolo26n_pose_v1_ft_manual_nopad`)뿐.

### 환경
- 신규 conda env **`pallet-yolo26`** = `pallet-pose` clone + `ultralytics 8.4.60`(yolo26 지원). 기존 `pallet-pose`(8.0.120) 보존.
- 이유: 기존 env의 ultralytics 8.0.120은 yolo26 로드 불가(`C3k2` 모듈 없음). GPU=RTX 3080 10GB.
- 경고: albumentations 버전차로 `ImageCompression quality_range` 비활성(부차 aug, 핵심 mosaic/scale은 ultralytics 자체라 영향 없음).

### 결과
- B 학습 완료: 31 epoch, best@epoch1 (early stop). A도 best@epoch7 — 둘 다 219장 작아 즉시 수렴.
- B best.pt: `runs/pose/challenge/weights/yolo26n_pose_v1_ft_manual_nopad/weights/best.pt`
  - (주의: project 상대경로라 `challenge/weights`가 아닌 `runs/pose/...` 아래 저장됨. A는 Windows 절대경로라 `challenge/weights`에 있음.)

## A/B 엄밀 비교 — holdout 재학습 + crop 평가

### 방향 (사용자 제안 채택)
- forklift 수동 라벨링 대신 **기존 GT를 crop해 truncation 인위 생성**. GT 좌표 known → crop offset 보정으로 화면 밖 keypoint도 정확 평가.

## A/B truncation 강건성 평가 파이프라인 (3d-expert)

### 신규 파일
- `challenge/scripts/eval_ab_crop.py` — crop level(0/1/2)×{A,B} 평가. argparse 재현 지원.
  - GT keypoint/6D 로드, A=100px reflect pad+(-100)shift / B=원본 추론.
  - crop 생성: GT 화면밖 코너 수로 level1(1~2)/level2(3~4) 자동 탐색, offset 으로 GT 보정.
  - multi-instance 시 GT centroid 최근접 detection association (한 frame 2팔레트 오선택 방지).
  - PnP = `scripts/self_training/pnp_solver.py` (EPnP+RANSAC), conf≥0.5 점만, 6점↑.
  - 지표: reproj(in/all, mean+median) · PnP성공% · ADD(mean+median) · 5cm5%.

### 좌표/convention 검증 (실행 전 1프레임 눈검증 완료)
- keypoint 순서 = `annotate_pnp` camera-facing(0~3 near,4~7 far,8 cen) = GT projected_cuboid.
- PnP 3D dims = **1.1×1.3×0.11m** (annotate 실측). config/default.yaml 1.1/1.1/0.15(KS) 아님.
- GT 6D = JSON pose_transform (cam 모델로 풀려 reproj median 2.3px). night08/09 는 stored
  pose 가 dim-swap → 두 dims 후보 중 reproj 더 맞는 쪽 자동 선택. -1.0 sentinel 마스킹.

- mean 은 multi-pallet/심한 occlusion 소수 frame skew → median 이 robust.
- 5cm5° 는 level1+ 에서 0 (flat 팔레트 광축 depth PnP 약제약 — task 특성, 버그 아님).

### 결론
- **level2 심한 truncation 에서 A(padding) 압승**: PnP 76.2% vs 45.2%, reproj_all med 21 vs 31.5px.
  A 가 화면밖 코너까지 9점 회귀 → PnP 6점 충족. B 는 detection 만 약간 높고 유효점 부족으로 PnP 붕괴.
- level0/1 두 모델 거의 동등. 차이는 truncation↑ 에서 벌어짐.
- **truncation 잦은 forklift 실배포에는 A(padding) 권장.**

## crop-aug truncation finetune (cropaug 모델 신규 학습 + 평가)

### 데이터 합성 (좌/우 측면 잘림 위주)
- 신규 스크립트 `challenge/scripts/gen_truncation_crops.py`: manual GT(capture* + forklift)를 crop+resize(640×480, 4:3 유지, 합성 fill 없음)해 truncation 샘플 증식. padding은 손대지 않음(이후 `convert --pad 100`).
- **cut 방향 측면 위주** (`CUT_WEIGHTS`): L/R ~55%, B계열 ~30%, T계열 ~5%. 사유: 실제 카메라는 좌우 패닝 → 측면 잘림이 현실적, top 잘림은 비현실적이고 degenerate(얇은 띠) 주범.
- **degenerate 필터**: in-image kp ≥5 AND visible-bbox 면적 ≥10% AND min(w,h) ≥50px.
- 산출: real `challenge/data/truncation_crops/` 485장(side 75%, top 5.8%, reject 17), synth(mixed_v8 200프레임) `challenge/data/truncation_crops_synth/` 394장(side 71%, top 3%).

### 데이터셋 (`yolo_pose_cropaug_padded`, pad=100)
- train **1007** = 원본 manual 177 + forklift 32 + real crop 404 + synth crop 394 (real 비중 61%).
- holdout 42 frame 파생 real crop **81개 제외** (leakage 방지). val 42 = A(`yolo_pose_manual_padded`)와 **동일** holdout (crop 미포함, 비교 일관성).
- yaml: `challenge/yolo_pose/data_cropaug_pad.yaml`. 좌표 1049줄 전부 [0,1] 정상.

### 학습 (`ft_cropaug.sh`)
- A와 동일 하이퍼파라미터, **epochs 60→120 / patience 30→40**만 변경. base=`yolo26n_pose_v1`.
- 결과: **best epoch 108**, holdout val pose mAP50-95 = **0.6417**. 약 18분(epoch당 ~9.2s, RTX 3080).
- weight: `runs/pose/challenge/weights/yolo26n_pose_v1_ft_cropaug/weights/best.pt`.

### 평가 — cropaug vs A(ft_pad_ho), 둘 다 pad 추론 (`eval_ab_crop.py --pad_b` 플래그 신규 추가)
`challenge/data/ab_crop_eval_cropaug/` — crop level 0/1/2, holdout 42. (A열=cropaug, B열=pad_ho)

| lvl | 지표 | cropaug | pad_ho(A) | 판정 |
|-----|------|---------|-----------|------|
| 0 | det% / PnP% | **100% / 90.5%** | 92.9% / 83.3% | cropaug ↑ |
| 0 | ADD med / 5cm5° | 130mm / 16.7% | **103mm / 23.8%** | pad_ho ↑ (clean 정밀도 소폭 후퇴) |
| 1 | det% / reproj_all med | **100% / 13.4px** | 95.2% / 15.5px | cropaug ↑ |
| 2 | **det% / PnP%** | **100% / 83.3%** | 85.7% / 76.2% | **cropaug 뚜렷 ↑ (truncation)** |
| 2 | reproj_in med | **8.2px** | 9.7px | cropaug ↑ |

- **결론**: crop-aug가 목표대로 **truncation 검출률·PnP 성공률을 뚜렷이 개선**(L2 det 85.7→100%, PnP 76.2→83.3%). 검출률은 전 레벨 100%로 안정화. trade-off: clean(L0) 케이스 pose 정밀도(ADD/5cm5°)는 소폭 후퇴 — synthetic+truncation 비중에 의한 것으로 추정(n=42라 노이즈도 있음).
- ADD/5cm5° 절대값이 전반적으로 나쁜 건 monocular scale 모호성(기존 이슈)으로, reproj median(~9px)이 keypoint 품질의 더 깨끗한 신호.

## crop-aug v2 — palletobj(진짜 스캔 팔레트 ~20k) crop 추가

### 배경
- 사용자 지적: synthetic crop을 mixed_v8(남의 팔레트)에서 떴는데, 실제로는 `challenge/data/training/v1+v2 = palletobj`(사용자 실제 팔레트, **19,991장**, frame_meta에 `truncation_applied` 포함)가 있음. base 모델 `yolo26n_pose_v1`은 palletobj+mixed_v8(`challenge/data/yolo_pose`, 27,552장)로 pretrain됐지만, v1 finetune엔 palletobj 직접 미포함.
- 결정: palletobj에도 **동일 측면-cut 증강** 적용해 추가, mixed_v8 crop은 유지.

### 데이터/학습
- palletobj 400프레임 → 측면-cut crop **748장** (side 77.9%, top 3.1%, reject 52). `challenge/data/truncation_crops_palletobj/`.
- v2 데이터셋 `yolo_pose_cropaug_v2_padded`: train **1755** = v1 구성(1007) + palletobj crop 748. real(실사진) 35% / synthetic 65%. val 42 동일.
- `ft_cropaug_v2.sh` (동일 설정). best **epoch 91**, clean-val mAP50-95 **0.6313** (v1 0.6417 대비 소폭↓).
- weight: `runs/pose/challenge/weights/yolo26n_pose_v1_ft_cropaug_v2/weights/best.pt`.

### 3자 비교 (eval_ab_crop, holdout 42, 모두 pad 추론)

| lvl | 지표 | pad_ho(A) | cropaug v1 | **cropaug v2** |
|-----|------|-----------|-----------|------------|
| 0 | det% / PnP% | 92.9 / 83.3 | 100 / 90.5 | **100 / 92.9** |
| 0 | reproj_in med | 8.9 | 8.9 | **8.6** |
| 0 | 5cm5° / ADD med | **23.8% / 103** | 16.7% / 130 | 11.9% / 151 |
| 1 | det% / reproj_all med | 95.2 / 15.5 | 100 / **13.4** | 100 / 14.7 |
| 2 | det% / PnP% | 85.7 / 76.2 | **100 / 83.3** | **100 / 83.3** |
| 2 | reproj_in med | 9.7 | **8.2** | 8.5 |

### 결론
- **truncation 검출·PnP 강건성은 v1≈v2 >> pad_ho** (목표 달성). palletobj 추가는 L0 PnP(90.5→92.9%)·reproj median 소폭 개선.
- **palletobj 추가가 v1을 뚜렷이 능가하진 않음.** 기대했던 clean(L0) 정밀도 회복은 안 됨 — 오히려 5cm5°(16.7→11.9%)·ADD(130→151) 더 후퇴.
- 원인 추정: clean 정밀도 후퇴는 **palletobj 모델 정체성 문제가 아니라 synthetic 비중(65%)** 자체. palletobj도 렌더라 real-photo 외형 갭 존재. 단 5cm5°/ADD는 n=42 + monocular depth라 노이즈 큼(L0 통과 frame 수 적음), keypoint 품질의 더 깨끗한 신호인 **reproj median은 v2가 최선/동등**.
- **실배포 권장**: truncation 강건성만 보면 v1/v2 동급. clean 정밀도 우선이면 synthetic 비중을 낮춘(예: mixed_v8 제거 또는 palletobj/synth crop 축소) 재학습 권장.

## forklift mp4 cropaug 추론 비디오 (keypoint + PnP cuboid overlay)

### 작업
- cropaug 모델(`yolo26n_pose_v1_ft_cropaug/weights/best.pt`, ★v1)로 `data/outside/forklift_raw_20260528_163408.mp4`(640×480, 911f, 15fps) 추론 → keypoint + PnP cuboid wireframe overlay 비디오 생성.

### 신규 스크립트
- `challenge/scripts/infer_video_yolo.py` (argparse: `--weights --source --cam_k --pad --out --conf --kp_conf --fps`). 다른 모델/영상 재사용 가능.
- 핵심: `eval_ab_crop.py`의 검증된 로직을 그대로 import 재사용 — `predict()`(100px reflect pad → predict → (-pad) shift), `solve_pnp()`(EPnP+RANSAC, dims=PALLET_DIMS=(1.1,1.3,0.11)), `draw_cuboid()`, `EDGES`. keypoint 색은 `annotate_draw.KP_COLORS`. cam_K는 모듈 전역 `ev.CAM_K`/`ev.PAD`를 override해 주입.
- 프레임별: pad+shift 추론 → conf≥0.5 점만 PnP(6점↑) → keypoint(색+번호) + 노란 cuboid wireframe + HUD(frame, DET/NO DET, PnP OK/FAIL, kp수, z거리) → mp4 누적 + 대표 6프레임 png.

### 결과 (911 frame)

## DOPE crop+padding 증강 학습 파이프라인 (YOLO crop-aug 방식 이식)

목표: YOLO 트랙에서 검증된 truncation crop 증강을 DOPE 학습에 이식. 사용자 결정 — pretrain crop=synthetic(mixed_v8) only, ft=real GT 251장, ft는 2단계(원본 long→crop short), 학습까지 백그라운드 실행.

### Phase A — crop+padding 데이터 (3d-expert 위임)
- **핵심 문제**: DOPE belief map(50×50)은 입력 범위 내 keypoint만 Gaussian 표현 → crop 후 화면 밖 corner는 supervision 없음. `CreateBeliefMap`(`Deep_Object_Pose/common/utils_belief.py:137`)이 `p±2σ`가 범위 안일 때만 Gaussian을 칠함(clip/sentinel 아님, 단순 skip)이 원인.
- **해결**: offline padding 채택. 고정 pad=100(YOLO값)은 DOPE엔 부족(crop의 화면밖 거리 median 132/p90 311px) → **crop별 dynamic 대칭 pad + reflect border + 640×480 resize-back**. MARGIN_FRAC=0.06→**0.20**으로 올려 belief 커버리지 68%→**100%** 달성 (output=50,σ=4 → 변에서 16% 안쪽 필요).
- 신규 스크립트 `challenge/scripts/pad_truncation_crops.py` (워크플로: `gen_truncation_crops.py` crop → `pad_truncation_crops.py` pad+resize-back). `gen_truncation_crops.py`는 미수정 재사용(이미 NDDS json 출력).
- 생성 데이터: pretrain crop `challenge/data/truncation_crops_dope/pretrain/` **8,831장**(mixed_v8 9000, variants=1, 169 degenerate reject), ft crop `challenge/data/truncation_crops_dope/ft_real/` **485장**(real 251, variants=2).
- 검증: DOPE 로더 통과 시 화면밖이던 corner까지 belief 8/8 supervised 확인. overlay `truncation_crops_dope/{pretrain,ft_real}/_verify/*_padded.png`, 로더 belief `truncation_crops_dope/_loader_check/*_loader_belief.png`.

### Phase B — Pretrain (실행 중)
- `train_dope.sh --exp_name dope_cropaug_pretrain --train_dirs "mixed_v8_train truncation_crops_dope/pretrain"` scratch 60ep, 백그라운드.
- 로더 실제 로드 17,831장(=9000+8831, _verify는 json 없어 자동 제외 확인). Loss 정상 감소, NaN 경고 1회뿐(무해).
- **버그 수정**: `scripts/train_dope.sh`가 CRLF(\r) 줄바꿈이라 bash 실패 → `sed`로 LF 변환(264 CRLF 제거).

### Phase C — 2단계 ft (완료)
- 신규 `scripts/ft_dope_cropaug.sh`: stage1=real 원본 251장 90ep → stage2=real+crop(485) 30ep. base=pretrain.
- real GT 251 = 낮 pallet02~09,cad 129 + 야간 night04~09 90 + forklift gt_manual 32 (빈 폴더 night01/03·pallet01·forklift_20260528 제외).
- **epoch convention 실수 정정**: train.py(`train.py:380`)는 finetune 시 net_path의 epoch에서 이어받고 EPOCHS=**누적 목표**로 해석. 처음 EPOCHS=90/30으로 줘서 stage1이 30ep만 돌고 stage2는 시작점(91)>목표(30)라 0ep이 될 뻔. → stage1 EPOCHS=**150**(60+90), stage2 EPOCHS=**180**(150+30)으로 수정 후 재실행. (model_catalog의 "120 (60→120, 60ep ft)" 표기가 이 convention)
- 결과: stage1 25min(`final_net_epoch_0150.pth`), stage2 23min(`final_net_epoch_0180.pth`).

### Phase D-2 — eval convention 통일 (order-free PnP) + 이전 해석 정정 (3d-expert)

`scripts/data_prep/eval/evaluate_on_val.py` 수정 (백업 `.bak`, CLI 인터페이스 유지 → train_dope.sh 호환):
| 항목 | Before | After |
|------|--------|-------|
| keypoint 매칭 | same-index | order-free 48 automorphism best-match (centroid idx8 고정) |
| PCK | same-index | order-free (+ same-index를 `(si)` 병기) |
| PnP/reproj | PalletPnPSolver EPnP+RANSAC | `annotate_pnp.solve_pose` (auto-dim, 24-sym, **ITERATIVE**) |
| dims | 1.1/1.1/0.15 | **1.1/1.3/0.11** |
| 신규 | — | 2D corner error (PnP 무관 순수 keypoint 품질) |

핵심 근거: ①**flat pallet은 EPnP 발산** — GT 2D self-consistency가 EPnP 43px vs ITERATIVE 4.9px. ②모델 v4 convention은 GT Isaac 대비 **W↔D swap relabeling 포함** → proper rotation 4개만으론 corner 75px, **48 automorphism 전체**로 16px.

재평가 (synthetic val, max 200, order-free):
| 모델 | PCK@3/5/10 | (same-idx@3) | corner med | PnP% | reproj med |
|------|-----------|--------------|-----------|------|-----------|
| baseline_v8_A | 79/85/88 | 49 | 28px | 71% | 39px |
| cropaug_ft_s1 | 81/89/96 | 23 | 16px | 64% | 21px |
| cropaug_ft_s2 | 78/86/94 | 22 | 18px | 58% | 20px |

- **⚠️ 이전 Phase D 해석 정정**: "crop 모델이 clean PCK 낮음(trade-off)"은 **convention 버그였음**. same-index가 crop-aug를 특히 과소평가(s1/s2 PCK@3 23/22% → 실제 **81/78%**). order-free로 보면 **crop-aug가 clean에서도 baseline 우세** (corner 16/18 vs 28px, PCK@10 96/94 vs 88%, reproj 21/20 vs 39px). crop 증강은 truncation뿐 아니라 clean keypoint localization도 개선.
- reproj 20~40px는 noisy keypoint에 rigid PnP fit하는 정상적 amplification (median 기준 비교, mean은 outlier skew).
- **결론**: synthetic val 평가 이제 신뢰 가능. flat object 평가는 EPnP 금지·ITERATIVE 필수, 매칭은 48 automorphism order-free.
