## DOPE → YOLOv8n-pose 전환 (Jetson Nano 배포용)

### 배경

challengenight (60 ep, 219 manual GT) 까지 학습된 DOPE 모델이 RTX 3080 에서는 27 FPS 로 잘 도는데, **원본 Jetson Nano (2019, 4GB Maxwell, 0.5 TFLOPs)** 에서는 VGG-19 + 448 입력 조합으로 1 FPS 미만. DopeMobileNet (CoRL/models.py) 으로 갈아끼워도 head 가 무거워 7~12 FPS 한계. **목표 15~30 FPS** 달성을 위해 YOLOv8n-pose 로 전환 결정.

### 비교 근거

```
방법                          예상 FPS (Nano)   재학습   비고
─────────────────────────────────────────────────────────────────────
DopeMobileNet 224 fp16 TRT    7~10            필요    head 가 병목
DopeMobileNet 224 INT8        12~15           필요    정확도 손실
YOLOv8n-pose 320 fp16 TRT ★  25~35           필요    Ultralytics 공식 benchmark
YOLOv8n-pose 640 fp16 TRT     15~20           필요    더 정확
```

근거: Ultralytics 공식 Nano benchmark (yolov8n detection): PyTorch fp32 ~3 FPS → TRT fp16 ~25 FPS → INT8 ~40 FPS. Pose 는 detection 의 70~85% 로 가정.

### Run 시 시퀀스 stuck / d_forward 요동 버그 수정 (사후 fix)

사용자 보고: Run 누르면 forklift 가 약 90° 회전 후 멈춤, d_forward 화살표가 매 프레임 요동. 분석 결과 두 가지 핵심 버그.

#### 버그 1 — `sideShift=false` 무한 루프 (시퀀스 stuck 의 진짜 원인)

원본 FSM 흐름:
- ALIGN_YAW 가 `targetLateralYaw` (= 팔레트 정면에 직각인 방향) 으로 회전 → ALIGN_LATERAL → REDETECT
- REDETECT 에서 `psiPallet` 체크 → 만족 못 함 → ALIGN_YAW 재진입
- ALIGN_YAW 는 또 `targetLateralYaw` 체크 → 이미 만족 → 즉시 통과 → ALIGN_LATERAL → ...

결과: `sideShift=false` 모드에서 lateral 정렬 후 팔레트 정면 yaw 로 다시 회전할 단계가 없어 무한 루프. 사용자가 본 "90° 회전 후 멈춤" = forklift 가 yaw 80°→0° (`targetLateralYaw`) 까지 회전 후 STOP_INTERLOCK/ALIGN_LATERAL/REDETECT/ALIGN_YAW 무한 반복.

**수정** (`sim.js`):
- `ALIGN_YAW` 에 `alignYawPhase` (`"lateral"` | `"pallet"`) 상태 추가, hysteresis 로 진입 시점에 한 번만 결정
- Before: `target = sideShift ? targetPalletYaw : targetLateralYaw`
- After: phase=="pallet" (sideShift=true 또는 진입 시 이미 dLateral ≤ latTol) → `targetPalletYaw`; 아니면 `targetLateralYaw`
- 종료 조건도 phase 에 따라 분기: pallet phase → `enterStop("REDETECT")`, lateral phase → `enterStop("ALIGN_LATERAL")`
- ALIGN_YAW timeout 8s 추가 (영구 stuck 방지)
- `enterState` 에서 `alignYawPhase = undefined` 로 매 진입마다 재계산

이제 흐름: ALIGN_YAW(phase=lateral) → ALIGN_LATERAL → REDETECT (psiPallet 미만족) → ALIGN_YAW(phase=pallet, dLateral≈0 이미 정렬됨) → REDETECT (모두 만족) → APPROACH.

#### 버그 2 — d_forward 요동 (detection 매 프레임 resample)

원본 `DETECTION`/`REDETECT` case 가 매 tick 마다 `sampleDetection()` 호출 → `detection.pallet` 위치가 매 프레임 noise 만큼 점프 → `dForward = (fork - C_hat) · n` 도 함께 요동.

**수정** (`sim.js`):
- `DETECTION`: 0.5s 마다 1 회 sample 후 `stateTime = 0` reset (사용자 부분 fix 보존, 검증 완료)
- `REDETECT`: 진입 시 `f.redetectSampled = false`, 0.25s 후 1 회만 sample (`f.redetectSampled = true`)
- `enterState` 에서 REDETECT 진입 시 자동 sample 제거 (위 sampling 으로 통일)

#### 버그 3 — Detection sigma 가 tolerance 보다 크거나 같음

기본값:
```
detYawStd = 3.0°     vs psiTol = 2.0°      ← yaw 1 sample 자체로 psiTol 초과 가능
detXStd   = 0.05 m   vs latTol = 0.05 m    ← lateral 정렬 후에도 detection 이 흔들면 ALIGN_YAW 재진입
detForwardStd = 0.05 m
```

**수정** (`sim.js` + `index.html` 기본값):
- detYawStd: 3.0 → **1.0** deg
- detXStd: 0.05 → **0.02** m
- detForwardStd: 0.05 → **0.02** m

#### 버그 4 — `ALIGN_LATERAL` 의 `lateralStart` 중복 설정

`enterState("ALIGN_LATERAL")` 에서 한 번, case 진입 시 또 한 번 설정. lateralStart 가 0.05 → 첫 tick 에서 갱신되어도 무해하지만 의도 불명확. 진단 메시지 추가 (`f.stateTime > lateralTimeout()` 분기에 `log("ALIGN_LATERAL timeout ...")`).

### 수정 후 head-trace (초기 pose 기준)

```
START → DETECTION (accepted=true 즉시) → ALIGN_YAW(phase=lateral, target=0°)
  → fork.yaw 80°→0° 회전 (~1.5s)
  → STOP_INTERLOCK → ALIGN_LATERAL (heading=+x, dLateral -0.5→0, ~1.1s)
  → STOP_INTERLOCK → REDETECT (재sample, psiPallet ≈ 90° 미만족)
  → ALIGN_YAW(phase=pallet, target=-90°=facing -y) (~1.6s 회전)
  → STOP_INTERLOCK → REDETECT (모두 만족)
  → APPROACH (heading=-y, dForward 감소, ~1.3s)
  → STOP_INTERLOCK → INSERT_FORK (slow=0.16m/s, ~13s)
  → STOP_INTERLOCK → RAISE_FORK (1.5s)
  → REVERSE (2.0s) → DONE
```

총 예상 ~22s. 시퀀스 완주 가능.

## 25y_automatic_lifter-master — depth_cam FSM 재설계 (snapshot-based "한 번 보고 눈 감고 동작")

### 배경

기존 `depth_cam/calib/fsm/` 는 매 프레임 perception 값 (yaw / offset / dist_z) 을 직접 분기 평가하는 실시간 FSM. 문제:
- IMU rel_yaw 가 한참 부족한데 perception yaw 가 노이즈로 잠깐 tol 안에 들어와서 단계 점프
- 회전 중에 vision yaw 가 흔들리면 `_stb.stable()` 카운터가 리셋되어 종료 판정이 발산
- 사용자가 "한 번 보고 눈을 감고 동작" 원칙으로 재설계 요청

### 새 모델

`*_CHECK` 상태 (`YAW_CHECK`, `OFFSET_CHECK`, `DIST_CHECK`) 진입 시점에 perception 값을 1회 snapshot 으로 캡처 + 다음 cmd 시퀀스의 모든 파라미터 사전 compute → SequencePlan 캐싱. cmd 실행 상태 (`YAW_CORRECT_*`, `LATERAL_ROTATE_*`, `FORWARD_AFTER_*`, `*_BACK`, `ALIGN_FWD_ADJUST`, `ALIGN_BWD_ADJUST`, `INSERT`) 진행 중에는 perception 무시, IMU rel_yaw 누적 / timer 종료 신호만 점검.

### 변경 파일

## Lift_FSM_simulator_package — LATERAL chain d_lateral 부호 수정 + 시뮬 검증

### 배경

config + dynamics 정합 적용 후 chrome-devtools 로 시뮬 실행 시 forklift 가 pallet 의 **반대 방향** 으로 이동. 분석:

- 초기 fork (-0.5, -0.9, yaw=80°), pallet centroid (0, 2.05) → 시뮬 dLat = (forklift - C) · t = -0.5 (forklift 가 entry face 의 좌측)
- 올바른 동작: 우측 평행 이동 (x → 0) → 정렬된 heading=+y 에서 ROT_RIGHT 90° → heading=+x → FWD → x 증가
- 실제 동작: ROT_LEFT 90° → heading=-x → FWD → x 더 감소 (-0.5 → -2.35). 반대 방향.

원인: `enterOffsetNeedCorrect` 에서 align.py 의 d_lat 부호 분기 (`d_lat < 0 → ROT_RIGHT`) 를 그대로 옮겼는데, **시뮬 dLat 와 align.py d_lat 는 부호가 정반대**.

```
align.py / pose6d_adapter d_lateral = entry face 의 카메라 +X (pallet 의 forklift 우측 거리)
시뮬 dLateral = (forklift - C) · t                 (forklift 의 entry face 기준 +t 측 거리)

같은 상황: forklift 가 pallet 의 좌측에 있을 때
  카메라 d_lateral > 0 (pallet 이 forklift 의 우측에 있다는 의미)
  시뮬 dLateral < 0 (forklift 가 -t 측에 있다는 의미)
→ 부호 정반대
```

추가로 **align.py 자체도 PDF 그림 11 패널 4 와 부호 어긋남:**
- PDF: `d_lateral < 0 → 제자리좌회전` (좌측 이동)
- align.py: `if d_lat < 0: sub = "LATERAL_ROTATE_RIGHT"; exec("ROT_RIGHT")` (우측 이동)
- → PDF 가 옳다면 align.py 도 부호 버그. 실차 테스트로 어느 게 옳은지 확정 필요.

### 수정 (sim.js)

`enterOffsetNeedCorrect` 의 direction 부호 반전:
```js
// Before:  const direction = dLat > 0 ? -1 : +1;
const direction = dLat > 0 ? +1 : -1;   // +1: ROT_LEFT, -1: ROT_RIGHT
```

#### 7. INSERT 거리 식 통일 — fork tip 기준 + safety margin

발견: 기존 식 `total = dist_z + pocket_m` 이 **fork center 기준** 이라 fork tip 이 pallet 뒷면 통과 (시뮬 검증: fork tip 이 pallet back +0.28m 통과 후 멈춤).

```
tip_target_into = pallet_depth - INSERT_SAFETY_BACK_M
total = max(0, dist_z - FORK_CENTER_TO_TIP_M + tip_target_into)
sec = total / INSERT_FWD_MPS
```

- 침투 깊이 1.19m (depth 의 91%)

## PnP crash 디버깅 + control.py mock fallback

### 사용자가 보낸 디버깅 로그 분석 (`디버깅.txt` / `디버깅1.txt`)

main_rec.py 실행 중 두 가지 OpenCV crash:

**에러 1**:
```
cv2.error: (-215:Assertion failed) m.dims >= 2 in function 'cv::Mat::Mat'
  at pose6d_adapter.py:147, cv2.solvePnP(obj_pts, img_pts, K, D, flags=flag)
```
→ `dist_coeffs=None` 일 때 `D = np.zeros(5)` 가 **1D**. OpenCV solvePnP 는 (N,1) 2D 요구.

**에러 2**:
```
DLT algorithm needs at least 6 points for pose estimation from 3D-2D point correspondences.
(expected: 'count >= 6'), where 'count' is 5
```
→ DOPE 가 가끔 5 keypoint 만 검출 (1개 가려짐). 어댑터의 `visible.sum() >= 6 ? EPNP : ITERATIVE` 분기에서 5 points → ITERATIVE 호출 → 초기 rvec/tvec 없어 fail.

### 수정 (`pose6d_adapter.py` + `main_rec.py`)

```python
# pose6d_adapter.py
D = np.zeros((5, 1), dtype=np.float64)   # 2D 로 reshape
if n_vis >= 6:
    flag = cv2.SOLVEPNP_EPNP
else:
    flag = getattr(cv2, "SOLVEPNP_SQPNP", cv2.SOLVEPNP_EPNP)   # OpenCV 4.5+: 4-5 points 지원
try:
    ok, rvec, tvec = cv2.solvePnP(obj_pts, img_pts, K, D, flags=flag)
except cv2.error:
    return None

# main_rec.py
try:
    result6d = keypoints9_to_align_vars(kps9, camera_matrix, dist_coeffs=None)
except Exception as e:
    print(f"[Pose6D] keypoints9_to_align_vars error: {e}")
    result6d = None
```

검증 (smoke test):
- **5 visible** (이전 crash) → SQPNP fallback → `(167°, -1.87m, 6.44m)` 정상
- **6 visible** → EPnP 정상

### `control.py` 의 canlib DLL 부재 대응 (`Kvaser SDK` 없는 머신용)

내 머신에서 main_rec.py 실행 시 `canlib32.dll` 부재로 import 단계 crash. 사용자 머신에는 Kvaser SDK 설치되어 있어 영향 없지만, dev/CI 머신에서도 import + main loop 검증이 가능하도록 fallback 추가:

```python
try:
    from canlib import canlib, Frame
    _CANLIB_DLL_OK = True
except (ImportError, FileNotFoundError, OSError) as e:

```
