### 핵심 결론 (발표 메시지)

1. **Self-training R1 단일 라운드가 최적 — R1 → R2 는 over-iteration 으로 모든 트랙에서 성능 하락**.
   F5 의 "single-round real-only ft" 결론을 outside/night 도메인까지 확장 검증.

2. **R1 → R2 PL pool 은 크게 증가 (3~15배) 하지만 quality 향상 효과 X**.
   - indoor: 2 → 31 (15.5x)
   - outside: 167 → 514 (3.1x)
   - night: 105 → 973 (9.3x)
   PL 양 증가는 confirmation bias 의 증거 (R1 의 self-detection 이 PL 로 amplify).

3. **ransac_loo 필터가 ransac 보다 압도적**: 1289 PL ransac 실패 vs 167 PL ransac_loo 성공. quality vs quantity.

4. **outside-PL 학습이 가장 generic 향상**: R1_outside_loo 가 indoor(58.4), outside(39.5), night(33.3) 모두에서 우수. 도메인-specific 학습이 아닌 generic detection 향상으로 작용.

5. **도메인 갭 검증**: R0 의 frame-with-prediction 비율 (indoor 71%, outside 53%, night 29%) 이 도메인 갭 직접 보임. R1 학습 후 모든 도메인 향상.

2. **YOLO 비활성 메시지 화면 표시** — `print("ℹ️ YOLO perception 비활성")` + HUD line `[mode] DOPE only (YOLO perception 비활성)` 추가. 사용자가 의식 안 한 단어를 일부러 띄움. → 사용자: "왜 갑자기 yolo 들고와서 그러냐고"

3. **gate 한 단계씩 풀기** — min_kp 7→5 후 실행 → reproj 12 발견 → 8→12 → 실행 → 16.8 발견 → 12→50 ... 반복. → 사용자: "한번에 해결됐으면 좋겠는데"

4. **실행 전 영향 분석 부재** — 한 gate 풀어도 다음 gate 가 또 fail 한다는 걸 예상 안 함. 매번 실행→발견→수정. → 사용자: "실행하기 전에 문제 될거 있는지 생각해봐"

### 근본 원인

1. **사용자가 누적한 setting 을 reference 로 보지 않음** — `challenge/config/task.yaml`, `CLAUDE.md`, `_docs/`, `config/default.yaml` 에 사용자가 명시한 값이 정답인데, 임의 추측으로 변경. 사용자의 누적된 작업 결과를 무시.

2. **변경의 cross-impact 매트릭스 미작성** — gate / FSM / 시각화 / 추론 / 좌표계 / resource (RealSense USB, CAN) 영역이 서로 영향. 한 변경 전체 영향 미리 점검 안 함.

3. **사용자가 의식 안 한 영역에 텍스트 추가** — YOLO 비활성 안내 같은 부작용. 사용자 입장에서 "왜 갑자기 등장?" 헷갈림 유발.

4. **사용자 코드 흐름 미파악** — DOPE 추론 path 통째 교체할 때 `fsm.py:237` 의 SEARCH 가 `det_ok` 만 본다는 사실 미인지. 사용자가 image #30 "잡혔는데 안 넘어감" 으로 발견.

5. **resource lifecycle 추적 미흡** — 이전 main_rec process 가 RealSense USB 점유한 상태에서 새 process 띄움. Frame timeout 반복 후에야 process kill 필요 인지.

6. **빠른 fix → 실행 → 보고 반복** — 사용자에게 단계마다 GUI 확인 요청. 사용자 시간 낭비.
