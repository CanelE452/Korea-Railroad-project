## annotate_pnp v4 fix v3 — Method C (strict invariant in PnP candidate scoring)

### 배경
fix v1/v2 의 post-process permute 방식은 PnP solver 가 거꾸로 풀린 face-flip 해를 채택해도 permute 만 적용해서 wireframe 이 거꾸로 그려지는 케이스 발생.

사용자 지적: `challenge/data/capturepallet03_manual_gt/1778651569891693056.json` 의 manual_kps (사용자 클릭) 는 정확히 v4 컨벤션 (vertical FRONT face 의 4 corner 클릭) 인데 fix v2 wireframe 이 거꾸로. reproj=31.88px 자체가 너무 높음 → PnP R, t 가 잘못됨.

### 진단
- PnP candidate scoring 이 reproj_error 만 사용 → face-flip 거꾸로 해도 reproj 작으면 채택
- polygon area 기반 v4 permute 는 R, t 자체가 거꾸로면 area 도 거꾸로 → fix v2 가 진짜 버그

### 해결 (Method C, `challenge/scripts/annotate_pnp.py`)
`_solve_pose_single` 의 PnP candidate scoring 에 v4 invariant 직접 검증:
- **Near-far**: cam-frame Z of cuboid local 0~3 의 평균 < 4~7 의 평균 (위반 +10000)
- **Front-largest-vert-face**: image polygon area (image-clipped) 의 FRONT (0-1-2-3) 가 4 vertical face 중 최대 (위반 +5000)
- Click ordering (기존 +1000 * n_viol)
- reproj_error (기존)

`score = reproj + 1000*n_viol + 10000*nf_viol + 5000*area_viol`. score 최소 후보 선택.

추가 안전장치:
- **Degenerate solution reject**: `t[2] > 50 * fx * max(dims) / click_span` (cuboid 너무 멀리 collapse), `max_face_area < 0.05 * click_area` (microscopic projection)
- **Planar PnP ambiguity**: `cv2.solvePnPGeneric(SOLVEPNP_IPPE)` 의 2 해를 init 후보에 추가
- **`solve_pose` 의 dim 선택**: 110 vs 130 정면 후보 모두 full score 로 비교 (reproj 단독 X)

## annotate.py PnP fix v4 — gravity invariant 추가 (TOP/BOT flip 해결)

### 배경

fix v3 (`_solve_pose_single` 의 near_far + area + n_viol scoring) 적용 후, capturepallet03 frame 26 에서
초록 wireframe 가 위아래 flip 된 cuboid 채택 (proj 0,1,4,5 = 사용자가 TOP 으로 의도한 점들이 image
**아래쪽**으로 그려짐). 빨간 v4 warning 도 정확한 click 에 false-positive 로 떴음.

진단:
- PnP 가 사용자 click 에 reproj 작은 후보 채택 (reproj=24px)
- 그러나 그 candidate 의 cam-frame Y 평균: TOP {0,1,4,5}=+0.451 > BOT {2,3,6,7}=+0.341
  → BOT 이 cam-Y 더 작음 (=image 위쪽) → **위아래 flip 된 cuboid 해**

## annotate.py PnP fix v7 — `u<0` sentinel 버그 + extrapolated 점 가중 PnP

### 사용자 보고

같은 frame `capturepallet03/1778651573855324672` 에서 7 missing 케이스:
- 클릭 7 점 (0~6), 7 = missing
- 0 = `t` TWO-LINE 외삽 (image 좌측 밖, click=(-16.7, 317.5))
- 1, 2 = 직접 click
- 3 = `x` parallelogram 외삽 (image 안 좌측 가까이)
- 4, 5, 6 = 직접 click
- 결과: wireframe 가 image 우측 작은 영역에 몰림 (z=3.99m, bbox 244×59 px,
  reproj=20.14 px saved)

### 진단 (Step 1)

`verify_truncation_fix_v7_step1_diagnose.py` 로 6-face IPPE × 24 cube symmetry ×
LM refine candidate 모두 enumerate (dim A + dim B = 465 candidate):

- **strict-pass candidate 291/465 존재**
- best strict-pass: dim B (1.3 front), **z=3.56m, reproj_click=3.38px** (직접 click
  1,2,4,5,6 기준)
- saved (선택된) pose: z=3.99m, reproj_click=18.11px — 잘못된 candidate 채택
- 모든 candidate 의 err_all ≈ 285,716 px (균일) ← **단서**

### 근본 원인

`_reproj_err_dict` sentinel 검출 잘못됨:
```python
# v1~v6 BUG:
if u < 0:                     # 잘못! valid image-out projection 도 u<0
    errs.append(1e6); continue
```

`project_3d` 의 진짜 sentinel = `(u, v) == (-1, -1)` (cam.z ≤ 0). 사용자가
image 좌측 밖 점을 정확히 클릭한 경우, 정상 projection 의 u 도 음수 → 모든 candidate
의 2개 idx (0, 3) 가 1e6 처리 → `(1e6 + 1e6 + 5_small) / 7 ≈ 285,714` 균일 →
tie-break 망가짐 → degenerate (작은 cube, 큰 z) 채택.

이 버그는 모든 클릭이 image 안인 경우 발현 안 됨 → v1~v6 잠재. truncation 시만 발현.

### Fix v7

1. **Sentinel 정확화** (`_reproj_err_dict`)
   - `if u == -1.0 and v == -1.0` 등호 검사
   - weights 인자 추가 (외삽 점 weight 조정용)

2. **Weighted reproj** (`_solve_pose_single`)
   - `extrapolated_mask` 인자 추가, 외삽 점 weight=0.3, 직접 click weight=1.0
   - selection 시 외삽 점 신뢰도 하향

3. **Degenerate cuboid reject** (`_solve_pose_single`)
   - cuboid 8-corner screen bbox area < image area * 1.5% candidate 제외
   - "wireframe 한 점에 몰림" 버그 가드

4. **GUI 통합** (`annotate.py` + `annotate_io.py` + `annotate_draw.py`)
   - `State.extrap_mask` 필드 (length 9 bool, frame 시작 시 초기화)
   - `t` (TWO-LINE) / `x` (parallelogram) 입력 시 `extrap_mask[active] = True`
   - L-click 으로 덮어쓰면 `False` (사용자가 클릭으로 보정)
   - `draw_overlay` 가 외삽 점은 outlined (속 빈 원) + `{idx}*` 라벨
   - panel 도움말에 `t/x` 옆 `*` 표시
   - `update_pose` 가 `solve_pose(... extrapolated_mask, img_shape)` 호출
